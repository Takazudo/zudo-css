# Concept and Architecture

## The Problem

When you ask AI to generate UI components, the results are creative but inconsistent. Each generation session produces its own ad-hoc colors, spacing values, font choices, and layout patterns. There is no shared design language across the output.

Consider asking an AI to "make 10 card variations." Without constraints, you get:

- Card A uses `#3b82f6` blue with `16px` padding and `8px` border radius
- Card B uses `rgb(59, 130, 246)` with `1.5rem` padding and `12px` border radius
- Card C uses `hsl(217 91% 60%)` with `20px` padding and `0.5rem` border radius

All three are "blue cards with medium padding," but they share no actual values. Drop them into the same page and the inconsistency is immediately visible. Multiply this across headers, footers, navigation, hero sections, and forms — the result is a page that looks assembled from different websites.

The deeper problem: **without explicit rules, AI reinvents the playground concept every time.** Each session creates a fresh, one-off demo environment with its own conventions. There is no accumulation of decisions, no shared foundation, no way to iterate on previous work.

## The Solution

**CSS Playground** is a small React application that lives inside your project, dedicated to generating and previewing UI component variations under strict design token constraints.

Instead of letting AI freestyle, the playground enforces a controlled environment where every generated component must use the project's token system. The result: AI produces dozens of creative variations that all share the same visual language.

The playground rests on **three pillars** — each one a strategy documented in this site:

1. **[Tight Token Strategy](/docs/methodology/tight-token-strategy)** — Replace all default CSS framework values with a small, intentional set of semantic tokens. AI cannot use arbitrary values because they do not exist.
2. **[Three-Tier Color Strategy](/docs/color/three-tier-color-strategy)** — Organize colors into palette, theme, and component tiers. AI picks from theme-level semantic colors, not raw hex values.
3. **[Component First Strategy](/docs/methodology/component-first-strategy)** — Express all UI as components with utility classes. AI generates components, not loose CSS class names.

## Architecture

The CSS Playground architecture connects the three pillars into a single development environment:

```
┌─────────────────────────────────────────────────┐
│                 CSS Playground App               │
│                (React sub-package)               │
├────────────┬────────────────┬───────────────────┤
│  Tight     │  Three-Tier    │  Component First  │
│  Token     │  Color         │  Strategy         │
│  Strategy  │  Strategy      │                   │
├────────────┴────────────────┴───────────────────┤
│           Shared Token Configuration             │
│     (CSS custom properties / Tailwind theme)     │
└─────────────────────────────────────────────────┘
```

### Tight Token Strategy in the Playground

The playground inherits the project's [tight token strategy](/docs/methodology/tight-token-strategy). All Tailwind defaults are reset — `p-4`, `bg-gray-500`, and other default utilities do not exist. Only the project's semantic tokens (`hsp-sm`, `vsp-md`, `text-body`, etc.) are available.

When AI generates a component inside the playground, it can only use tokens that exist. Attempting to use `gap-4` or `bg-blue-500` fails at build time. This is the constraint that forces consistency.

### Three-Tier Color in the Playground

The playground uses the project's [three-tier color system](/docs/color/three-tier-color-strategy). AI-generated components reference theme-tier tokens like `--theme-accent` and `--theme-fg`, not raw palette values.

The playground adds an **interactive color palette panel** — a UI where you can adjust palette values and watch every component update live. Because all components reference theme tokens, and theme tokens reference palette values, tweaking the palette propagates through the entire component gallery instantly.

### Component First in the Playground

Every piece of UI in the playground is a [component](/docs/methodology/component-first-strategy) — a React function that composes utility classes. AI does not generate loose CSS files or class names. It generates components that follow the project's conventions.

## How It Works

The development workflow with CSS Playground follows a cycle:

1. **Prompt** — You describe what you want to AI: "Generate 40 header menu variations using our token system"
2. **Generate** — AI creates component files inside the playground, using only available tokens
3. **Preview** — Open the playground in the browser to see all variations rendered in a gallery
4. **Tune** — Use the color palette panel to tweak the palette and see all components update
5. **Select** — Pick the variations you like, iterate on them with more AI prompts
6. **Extract** — Move selected components from the playground into the main application

## Project Structure

A typical project with CSS Playground has this layout:

```
my-project/
├── src/                      # Main application
│   ├── components/           # Production components
│   ├── styles/
│   │   └── tokens.css        # Shared token definitions
│   └── ...
├── css-playground/           # Playground sub-package
│   ├── package.json          # Separate dependencies (React, Vite)
│   ├── src/
│   │   ├── main.tsx          # Playground entry point
│   │   ├── App.tsx           # Gallery layout + color panel
│   │   ├── components/       # AI-generated component variations
│   │   │   ├── headers/      # Header variations
│   │   │   ├── footers/      # Footer variations
│   │   │   ├── cards/        # Card variations
│   │   │   └── ...
│   │   └── styles/
│   │       └── tokens.css    # Imports shared tokens + playground overrides
│   └── index.html
├── tailwind.config.ts        # Shared Tailwind config (tight tokens)
└── package.json
```

Key points:

- The playground is a **separate sub-package** with its own `package.json` and dev server
- It **imports the same token configuration** as the main application
- AI-generated components are organized by category inside `css-playground/src/components/`
- The playground has its own entry point that renders a gallery view of all components

## The Color Palette Panel

One of the most valuable features of the playground is the interactive color palette panel. This is a React component that:

- Reads all palette-tier CSS custom properties
- Renders color pickers for each palette value
- Updates CSS custom properties in real time when you adjust a color
- All components in the gallery update instantly because they reference theme tokens that chain to palette values

## When to Use

### Good Fit

- **Projects with a design system** — You already have (or want to define) a set of design tokens
- **AI-assisted development** — You use AI tools to generate UI components and want consistent output
- **Rapid prototyping** — You need to explore many design variations quickly while staying on-brand
- **Team projects** — Multiple developers (or AI agents) generating components need a shared visual language
- **Component library development** — Building a library where every component must use the same token system

### Not Needed

- **Small single-page sites** — If you have a handful of components, a playground adds unnecessary complexity
- **Projects without a component framework** — The playground assumes React (or similar) components
- **One-off prototypes** — If consistency across components does not matter, unconstrained generation is fine
- **Static content sites** — If the site is mostly text with minimal interactive UI, token enforcement has less impact
