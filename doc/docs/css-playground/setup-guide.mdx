# Setup Guide

## The Problem

Setting up a CSS Playground from scratch every time is error-prone. Without a standardized process, each setup produces a slightly different structure — different folder layouts, different token imports, different build configurations. The whole point of the playground is consistency, but if the playground itself is inconsistent across projects, the benefit is undermined.

AI agents compound this problem. When asked to "set up a CSS playground," an AI will invent its own interpretation every time — different directory names, different entry points, different approaches to token sharing. You need a prescriptive setup that tells AI **exactly** what to build.

## The Solution

A standardized setup process that produces the same playground structure in every project. Follow these steps to add a CSS Playground to any project that uses a component framework and a token-based CSS system.

### Step 1: Directory Structure

Create the playground as a sub-package in your project root:

```
css-playground/
├── package.json
├── index.html
├── vite.config.ts
├── tsconfig.json
├── src/
│   ├── main.tsx
│   ├── App.tsx
│   ├── styles/
│   │   ├── tokens.css          # Token imports + playground overrides
│   │   └── playground.css      # Gallery layout styles
│   └── components/
│       ├── headers/
│       │   ├── HeaderVariation01.tsx
│       │   ├── HeaderVariation02.tsx
│       │   └── index.tsx        # Barrel export
│       ├── footers/
│       ├── cards/
│       ├── heroes/
│       └── navigation/
└── README.md
```

### Step 2: Package Configuration

The playground is a standalone Vite + React app. It has its own dependencies but shares the token configuration with the main project:

```json
{
  "name": "@my-project/css-playground",
  "private": true,
  "type": "module",
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  },
  "dependencies": {
    "react": "^19.0.0",
    "react-dom": "^19.0.0"
  },
  "devDependencies": {
    "@types/react": "^19.0.0",
    "@types/react-dom": "^19.0.0",
    "@vitejs/plugin-react": "^4.0.0",
    "typescript": "^5.0.0",
    "vite": "^6.0.0"
  }
}
```

### Step 3: Token Connection

The playground must use the same tokens as the main project. There are two approaches depending on your CSS framework.

#### With Tailwind CSS (Tight Token Strategy)

If the main project uses the [tight token strategy](/docs/methodology/tight-token-strategy), the playground imports the same Tailwind configuration:

```css
/* css-playground/src/styles/tokens.css */
@import "tailwindcss";

/* Import the shared tight token theme from the main project */
@import "../../src/styles/theme.css";
```

The shared `theme.css` contains all the `@theme` resets and semantic token definitions. Both the main app and the playground reference this single source of truth.

#### With CSS Custom Properties

If the project uses plain CSS custom properties with the [three-tier color strategy](/docs/color/three-tier-color-strategy):

```css
/* css-playground/src/styles/tokens.css */

/* Import shared palette and theme tiers from the main project */
@import "../../src/styles/palette.css";
@import "../../src/styles/theme.css";

/* Playground-specific overrides (if needed) */
:root {
  --playground-gallery-bg: hsl(220 15% 96%);
  --playground-sidebar-bg: hsl(220 20% 14%);
}
```

### Step 4: Entry Point

The playground entry point is a standard Vite + React setup:

```html
<!-- css-playground/index.html -->
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>CSS Playground</title>
  </head>
  <body>
    <div id="root"></div>
    <script type="module" src="/src/main.tsx"></script>
  </body>
</html>
```

```tsx
// css-playground/src/main.tsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import App from './App';
import './styles/tokens.css';
import './styles/playground.css';

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>
);
```

### Step 5: Gallery App Shell

The `App.tsx` provides the gallery layout — a sidebar with the color palette panel and a main area that renders all component variations:

```tsx
// css-playground/src/App.tsx
import { useState } from 'react';
import { headers } from './components/headers';
import { cards } from './components/cards';

type Category = 'headers' | 'cards' | 'footers' | 'heroes';

export default function App() {
  const [activeCategory, setActiveCategory] = useState<Category>('headers');

  const categories: Record<Category, React.FC[]> = {
    headers,
    cards,
    footers: [],
    heroes: [],
  };

  const components = categories[activeCategory];

  return (
    <div className="playground">
      <aside className="playground__sidebar">
        <h2>CSS Playground</h2>
        <nav>
          {Object.keys(categories).map((cat) => (
            <button
              key={cat}
              onClick={() => setActiveCategory(cat as Category)}
              className={cat === activeCategory ? 'active' : ''}
            >
              {cat}
            </button>
          ))}
        </nav>
        {/* Color palette panel goes here */}
      </aside>
      <main className="playground__gallery">
        {components.map((Component, i) => (
          <div key={i} className="playground__item">
            <Component />
            <span className="playground__label">Variation {i + 1}</span>
          </div>
        ))}
      </main>
    </div>
  );
}
```

### Step 7: Add to Root Package Scripts

Add convenience scripts in the project root `package.json`:

```json
{
  "scripts": {
    "playground": "cd css-playground && npm run dev",
    "playground:build": "cd css-playground && npm run build"
  }
}
```

## AI Instructions File

The most critical piece: **a CLAUDE.md (or equivalent AI instruction file) in the playground directory** that tells AI exactly what rules to follow when generating components:

```markdown
# CSS Playground Instructions

## Rules for Generated Components

1. ONLY use tokens defined in tokens.css — no arbitrary values
2. Every component is a React function component
3. Use utility classes from the project's Tailwind config
4. No custom CSS classes — use component-first strategy
5. No inline styles except CSS custom property references
6. All colors must come from theme-tier tokens
7. All spacing must use hsp-* or vsp-* tokens
8. Export each variation as a default export from its own file
9. Add each new variation to the category barrel export
```

This file is what prevents AI from reinventing the playground. Without it, every AI session starts from scratch with its own conventions. With it, AI knows exactly what tokens exist, what patterns to follow, and what structure to use.

## The Color Palette Panel

The interactive color palette panel is what makes the playground a visual tool, not just a code generator. Here is a minimal implementation:

```tsx
// css-playground/src/components/ColorPalettePanel.tsx
import { useCallback } from 'react';

interface PaletteColor {
  name: string;
  property: string;
  defaultValue: string;
}

const paletteColors: PaletteColor[] = [
  { name: 'Accent', property: '--palette-accent', defaultValue: '#3b82f6' },
  { name: 'Accent Light', property: '--palette-accent-light', defaultValue: '#eff6ff' },
  { name: 'Foreground', property: '--palette-fg', defaultValue: '#1e293b' },
  { name: 'Muted', property: '--palette-fg-muted', defaultValue: '#64748b' },
  { name: 'Background', property: '--palette-bg', defaultValue: '#ffffff' },
  { name: 'Border', property: '--palette-border', defaultValue: '#cbd5e1' },
];

export default function ColorPalettePanel() {
  const handleChange = useCallback(
    (property: string, value: string) => {
      document.documentElement.style.setProperty(property, value);
    },
    []
  );

  return (
    <div className="palette-panel">
      <h3>Color Palette</h3>
      {paletteColors.map(({ name, property, defaultValue }) => (
        <label key={property}>
          <input
            type="color"
            defaultValue={defaultValue}
            onChange={(e) => handleChange(property, e.target.value)}
          />
          {name}
        </label>
      ))}
    </div>
  );
}
```

This works because of the [three-tier color strategy](/docs/color/three-tier-color-strategy): components reference theme tokens → theme tokens reference palette tokens → the panel updates palette tokens. The entire chain updates in real time.

## Checklist

Use this checklist when setting up a new CSS Playground:

- [ ] Create `css-playground/` directory at project root
- [ ] Initialize `package.json` with React and Vite dependencies
- [ ] Set up `tokens.css` that imports the main project's token configuration
- [ ] Create `index.html` and `main.tsx` entry point
- [ ] Create `App.tsx` with gallery layout and category navigation
- [ ] Add `ColorPalettePanel` component for interactive color tuning
- [ ] Create component category folders (`headers/`, `cards/`, `footers/`, etc.)
- [ ] Add barrel exports (`index.tsx`) to each category
- [ ] Add convenience scripts to root `package.json`
- [ ] Create `CLAUDE.md` with AI generation rules specific to the project's tokens
- [ ] Verify the playground builds and renders correctly

## When to Use

Set up a CSS Playground when:

- **Starting a new project** — Establish the playground alongside the initial token system
- **Adding design exploration capability** — When the team wants to use AI for rapid UI variation generation
- **Building a component library** — The playground serves as both a development tool and a living style guide
- **Onboarding AI to an existing project** — When you want AI to generate components that match existing conventions
