# Usage Patterns

## The Problem

Without a structured approach, asking AI to "make a header" produces a one-off result. Each request is disconnected — no shared tokens, no consistent spacing, no visual cohesion across generated components. Ask for a card next and the AI picks entirely different colors, spacing values, and sizing conventions. The results _look_ like they came from different projects.

This gets worse at scale. If you generate 10 button variations, 5 card layouts, and 3 navigation bars in separate prompts, you end up with 18 components that share nothing. Merging them into a cohesive page requires manually re-aligning every spacing value, every color, every font size — defeating the purpose of AI-assisted generation.

The root cause: the AI has no constraints. Without explicit rules about tokens, colors, and component structure, it invents its own system for every request.

## The Solution

Use a **structured prompting workflow** that tells the AI exactly which design tokens to use, how to structure color, and how to output components. The CSS Playground enforces these rules so that every generated component shares the same visual language.

The workflow has three phases:

1. **Constrain** — Define the token system (spacing, color, sizing) before generating anything
2. **Generate** — Request multiple component variations in a single prompt, referencing the token system
3. **Iterate** — Preview results in the playground, refine constraints, generate more

## Prompt Patterns

### Pattern 1: Batch Variation Generation

Instead of asking for one component at a time, request many variations in a single prompt. This forces the AI to think about differentiation within a shared system.

**Prompt example:**

> Generate 6 button variations using these constraints:
> - Spacing: hsp-xs (12px) horizontal padding, vsp-2xs (4px) vertical padding
> - Colors: use --theme-accent, --theme-fg, --theme-bg, --theme-muted
> - Structure: React components with utility classes only, no custom CSS class names
> - Each variation must be visually distinct but use ONLY the provided tokens

### Pattern 2: Specifying Constraints Explicitly

The more explicit your constraints, the more consistent the output. Always specify:

1. **Token system** — which spacing tokens to use (hsp-\*/vsp-\*)
2. **Color tier** — which theme tokens are available
3. **Output format** — React components with utility classes
4. **What NOT to do** — no arbitrary values, no custom CSS classes

### Pattern 3: Component Type Galleries

Request a gallery of a single component type. This is the most common playground pattern — generating many variations of one thing to explore the design space.

## Iteration Workflow

The playground workflow is a loop, not a one-shot process:

1. **Generate** — Request N variations of a component type with explicit token constraints
2. **Preview** — View all variations side by side in the playground
3. **Refine** — Identify what works, adjust constraints (tighten spacing, try different theme mappings)
4. **Generate more** — Request additional variations with updated constraints
5. **Extract** — Pick the best variations and move them into production

## Rules for AI

When generating components for the CSS Playground, AI agents must follow these rules exactly:

1. **Always use the project's semantic spacing tokens** — `hsp-*` for horizontal spacing, `vsp-*` for vertical spacing. Never write arbitrary pixel values like `padding: 18px 22px`.

2. **Always use three-tier color** — Components reference `--theme-*` tokens (Tier 2), theme tokens reference `--palette-*` tokens (Tier 1). Never use raw color values like `hsl(250 60% 45%)` directly in components.

3. **Component first** — Every variation is a React component with utility classes. No custom CSS class names, no separate stylesheets, no CSS modules.

4. **No arbitrary values** — Every spacing, color, size, and border-radius must come from a defined token. If a value is not in the token system, it is not allowed.

5. **Self-contained variations** — Each generated variation must work independently. No shared mutable state between variations.

## When to Use

- **Rapid UI exploration** — Generate 10–20 variations of a component to explore the design space quickly
- **Design system validation** — Test whether your token system is expressive enough by seeing how many distinct variations it can produce
- **Building a component library** — Use the playground to generate, preview, and select the best component variants for your library
- **AI-assisted development** — Any time you use AI to generate UI components, the structured prompting workflow prevents visual drift
- **Client presentations** — Generate multiple options quickly to present design directions, all guaranteed to be visually consistent
